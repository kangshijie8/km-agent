# Kunming Agent CLI 阻塞问题排查指南

## 问题概述

**现象**: km agent 在运行过程中卡住，无法响应新消息，30秒超时后显示 "Agent thread is stuck and did not terminate within 30s after interrupt"

**根本原因**: Windows 平台上，当 agent 线程阻塞在 httpx 的 C 扩展（SSE 流读取）时，会持有 Python GIL，导致主线程的 `join()` 无法获取 GIL，形成死锁。

#### 修复 3: 使用 ThreadPoolExecutor 实现可中断的 stream 迭代

**位置**: line 4872-4957

**核心修改**:

```python
# 使用 ThreadPoolExecutor 包装 stream 迭代
import concurrent.futures as _futures

def _iter_stream_with_interrupt():
    """Generator that yields stream events but checks for interrupts."""
    for evt in stream:
        if self._interrupt_requested:
            raise InterruptedError("Stream interrupted by user request")
        yield evt

# 在后台线程执行迭代
with _futures.ThreadPoolExecutor(max_workers=1) as _executor:
    _future = _executor.submit(lambda: list(_iter_stream_with_interrupt()))
    
    # 主线程轮询检查中断
    while not _future.done():
        if self._interrupt_requested:
            _future.cancel()
            self._anthropic_client.close()
            raise InterruptedError("Agent interrupted during Anthropic stream")
        time.sleep(0.1)
    
    _all_events = _future.result(timeout=1.0)

# 处理收集到的事件
for event in _all_events:
    # ... 处理逻辑 ...
```

### 2026-04-16 第四次修复

**问题**: `InterruptedError` 被 `_call()` 函数的异常处理器捕获

**症状**: 即使 `_call_anthropic()` 正确抛出 `InterruptedError`，agent 线程仍然卡住

**根本原因**: `_call()` 函数内部的 `except Exception as e:` 捕获了 `InterruptedError`，将其当作普通网络错误处理，导致中断信号被吞掉

**修复文件**: `run_agent.py`

**修复位置**: line 4978-4990

**修复内容**:
```python
def _call():
    # ...
    for _stream_attempt in range(_max_stream_retries + 1):
        try:
            if self.api_mode == "anthropic_messages":
                result["response"] = _call_anthropic()
            else:
                result["response"] = _call_chat_completions()
            return  # success
        except InterruptedError:
            # CRITICAL FIX 2026-04-16: Re-raise InterruptedError immediately
            # so the outer interrupt logic can handle it properly.
            raise
        except Exception as e:
            # ... 网络错误重试逻辑 ...
```

**关键教训**: `InterruptedError` 必须在异常处理链中优先处理，不能被通用的 `Exception` 捕获

### 2026-04-16 第五次修复

**问题**: `ThreadPoolExecutor` 的 `shutdown(wait=True)` 导致 agent 线程卡住

**症状**: 即使 stream 迭代被 `future.cancel()` 取消并抛出了 `InterruptedError`，agent 线程在退出 `with ThreadPoolExecutor(...) as executor:` 上下文时，会调用 `executor.shutdown(wait=True)`。如果工作线程仍然阻塞在 httpx C 扩展中，`shutdown(wait=True)` 会永远等待，导致 `agent_thread.join(timeout=30)` 超时并记录 `AGENT_THREAD_STUCK`。

**根本原因**: `with ThreadPoolExecutor(...)` 在 `__exit__` 时默认调用 `shutdown(wait=True)`。对于已经开始执行但阻塞在不可中断 I/O 的工作线程，`cancel()` 无效，`shutdown(wait=True)` 会无限期等待。

**修复文件**: `run_agent.py`

**修复位置**: line 4074, 4217, 4732, 4964

**修复内容**: 将 4 处流式调用的 `with ThreadPoolExecutor(max_workers=1) as _executor:` 改为手动创建和 `shutdown(wait=False)`：
```python
_executor = _futures.ThreadPoolExecutor(max_workers=1)
try:
    _future = _executor.submit(lambda: list(_iter_stream_with_interrupt()))
    ...
    _all_events = _future.result(timeout=1.0)
finally:
    _executor.shutdown(wait=False)
```

### 2026-04-16 第六次修复

**问题**: 429 rate limit 退避期间不响应中断

**症状**: 当模型返回 429 时，agent 会 `time.sleep(backoff)` 最长 300 秒。如果用户在此期间发送新消息，`interrupt()` 设置了 `_interrupt_requested=True`，但 agent 仍然 sleep 完整个退避时间。

**根本原因**: `_handle_provider_error()` 中的 429 退避使用了一次性 `time.sleep(backoff)`，没有检查中断标志。

**修复文件**: `run_agent.py`

**修复位置**: line 4470-4474

**修复内容**: 将一次性 sleep 改成分段 sleep，每 200ms 检查一次中断：
```python
_slept = 0.0
while _slept < backoff:
    if self._interrupt_requested:
        break
    time.sleep(min(0.2, backoff - _slept))
    _slept += 0.2
```

## 排查方法

### 1. 确认阻塞位置

检查 `interrupt_debug.log` 文件：
```bash
type C:\Users\18687\.kunming\interrupt_debug.log
```

### 2. 确认 API 模式

查看当前使用的 API 模式：
- OpenAI 模式: `api_mode == "openai"`
- Anthropic 模式: `api_mode == "anthropic_messages"`
- Codex 模式: `api_mode == "codex_responses"`

### 3. 检查客户端关闭逻辑

在 `interrupt()` 方法中，确保对应模式的客户端关闭代码已执行：
- OpenAI: `_close_request_openai_client()`
- Anthropic: `anthropic_client.close()`

### 4. 验证 stream 迭代方式

检查 `_call_anthropic()` 或 `_call_chat_completions()` 是否使用了可中断的迭代模式：
- 是否使用 ThreadPoolExecutor
- 是否有中断检查点
- 是否有硬超时保护

## 常见阻塞场景

### 场景 1: SSE 流读取阻塞

**症状**: Agent 在等待模型响应时卡住，不返回任何内容

**原因**: httpx 的 C 扩展阻塞在 `read()` 调用，持有 GIL

**解决**: 
- 确保 `interrupt()` 关闭底层 HTTP 连接
- 使用 ThreadPoolExecutor 包装 stream 迭代

### 场景 2: 工具调用阻塞

**症状**: 工具执行时卡住（如 terminal、browser）

**原因**: 工具内部阻塞，不检查 `_interrupt_requested` 标志

**解决**:
- 工具实现中添加中断检查点
- 使用超时机制

### 场景 3: 子 Agent 阻塞

**症状**: 使用 delegate 工具时子 agent 卡住

**原因**: 子 agent 的 interrupt 没有正确传播

**解决**:
- 确保 `interrupt()` 中调用 `child.interrupt()`
- 检查 `_active_children` 列表
