## 🔍 CLI阻塞问题深度排查报告

### 一、核心阻塞点定位

**阻塞发生在 `chat()` 方法中的主等待循环** ([cli.py:L6545-6589](file:///d:/kunming/cli.py#L6545-L6589))：

```
用户发送消息 → Enter key handler → _pending_input.put() → process_loop处理 → chat()调用
     ↓
[cli.py:L6545-6548] agent_thread.is_alive() 循环 + interrupt_queue.get(timeout=0.1)
     ↓
[cli.py:L6589] agent_thread.join(timeout=10) ← 这里会阻塞最多10秒
```

### 二、阻塞根因分析（为什么8轮修复都无效）

**根本原因：之前的修复都是"治标不治本"，没有识别出真正的阻塞链**

#### 阻塞链完整路径：

1. **[cli.py:L6545-6583] interrupt监控循环阻塞**
   - 原理：`while agent_thread.is_alive()` 循环每0.1秒poll一次
   - 问题：如果agent_thread正常执行（无interrupt），这个循环会一直等到agent_thread自然结束
   - **不是问题本身，但掩盖了真正的问题**

2. **[cli.py:L6589] agent_thread.join(timeout=10) 阻塞** 
   - 原理：Windows GIL下threading.Thread.join()可能被卡住
   - 之前修复：超时从30s降到10s，但仍然会阻塞10秒
   - **这是用户感知到的"发送消息后卡住10秒"的直接原因**

3. **[cli.py:L6515-6521] run_conversation() API调用阻塞** 
   - 原理：Anthropic/Codex SDK的`client.chat.completions.create()`是同步阻塞调用
   - 之前修复：timeout从900s降到300s（[git commit 14b0b1f6](file:///d:/kunming/cli.py#L6515)）
   - **这是长时间阻塞的根本原因**

4. **[cli.py:L6628] tts_thread.join(timeout=120) 阻塞**
   - 原理：TTS语音播放完成后join，超时120秒
   - **如果TTS卡住，会阻塞2分钟**
